Run Any LLM Locally With vLLM and Docker

March 2026 · 20 min read · Requires: NVIDIA GPU + Docker

Calling an API is easy — until you start worrying about what leaves your machine, how much it costs per token, or why there's a 30-second timeout on every request. Self-hosting changes that equation entirely. This guide walks through hosting any Hugging Face model locally using vLLM and Docker, so you get a production-grade OpenAI-compatible endpoint running on your own GPU.

What Is vLLM, and Why Should You Care?

vLLM is a fast inference engine for large language models. The original research behind it introduced PagedAttention — a technique that manages GPU memory the way operating systems manage RAM, preventing the fragmentation that normally wastes 20–40% of your VRAM on other frameworks. The practical upshot: higher throughput, more concurrent requests, and significantly faster time-to-first-token.

Beyond the raw performance, vLLM exposes an OpenAI-compatible REST API out of the box. That means any code you've already written against the OpenAI SDK works without modification — you just point it at localhost:8000 instead of api.openai.com. For local development and private deployments, this is hard to beat.

Who this is for: ML engineers who want a reproducible, low-friction way to host any HuggingFace model without writing custom serving code. If you've ever wasted an afternoon wiring up FastAPI around a transformers pipeline, this is the saner path.

Before You Start

You'll need the following in place before anything else:

  • An NVIDIA GPU with enough VRAM for your chosen model (7B models need ~16 GB; 70B models need 80 GB+)
  • Docker with the NVIDIA Container Toolkit installed
  • A Hugging Face account and a token if you're pulling gated models like Llama
  • Disk space — model weights are large; a 7B model is roughly 13–14 GB

Check what GPUs you have available by running:

bash 
nvidia-smi -L

If that command gives you a list of GPUs rather than an error, you're ready to move on. If CUDA isn't visible inside Docker, make sure you've installed the NVIDIA Container Toolkit and that your Docker daemon is configured to use it.

The Dockerfile — Build Once, Use Forever

The goal here is a single Docker image that can serve any model by passing a name at runtime. No rebuilding every time you want to try something new — just change an environment variable.

Create a file called Dockerfile and paste this in:

Dockerfile 
FROM nvcr.io/nvidia/vllm:26.01-py3

WORKDIR /app

# Hugging Face cache — MUST match the volume mount path
ENV HF_HOME=/models
ENV TRANSFORMERS_CACHE=/models
ENV TORCH_HOME=/models

# Configurable defaults — override at runtime with -e
ENV MODEL_ID=""
ENV SERVED_MODEL_NAME=""
ENV TENSOR_PARALLEL_SIZE=1
ENV MAX_NUM_SEQS=8
ENV MAX_MODEL_LEN=32768
ENV DTYPE=auto
ENV GPU_MEMORY_UTILIZATION=0.90
ENV QUANTIZATION=""
ENV EXTRA_ARGS=""

RUN pip install --no-cache-dir accelerate bitsandbytes

EXPOSE 8000

CMD ["bash", "-c", "\
  if [ -z \"$MODEL_ID\" ]; then \
    echo 'ERROR: MODEL_ID is not set. Pass it with -e MODEL_ID=...' && exit 1; \
  fi; \
  SERVED_NAME=${SERVED_MODEL_NAME:-$(basename $MODEL_ID)}; \
  QUANT_ARG=$([ -n \"$QUANTIZATION\" ] && echo \"--quantization $QUANTIZATION\" || echo \"\"); \
  vllm serve $MODEL_ID \
    --served-model-name $SERVED_NAME \
    --host 0.0.0.0 \
    --port 8000 \
    --tensor-parallel-size $TENSOR_PARALLEL_SIZE \
    --max-num-seqs $MAX_NUM_SEQS \
    --max-model-len $MAX_MODEL_LEN \
    --gpu-memory-utilization $GPU_MEMORY_UTILIZATION \
    --trust-remote-code \
    --dtype $DTYPE \
    $QUANT_ARG \
    $EXTRA_ARGS \
"]

Build the image:

bash 
docker build -t vllm-universal .

This takes a few minutes the first time. After that, you never rebuild it — you just swap the model at runtime.

Why set HF_HOME inside the Dockerfile? If you don't, the model downloads into the container's filesystem. When you delete the container, the 13 GB download goes with it. By pointing HF_HOME to /models and mounting your host cache there, the weights persist on disk between runs.

Running Your First Model

Let's spin up Mistral 7B Instruct. The pattern here is the same for almost any model — just swap the name.

bash 
MODEL="mistralai/Mistral-7B-Instruct-v0.3" && docker run -it --gpus all \
  -p 8000:8000 \
  -v ~/.cache/huggingface:/models \
  -e MODEL_ID="$MODEL" \
  --name $(basename "$MODEL") \
  --ipc=host \
  --ulimit memlock=-1 \
  --ulimit stack=67108864 \
  vllm-universal

A few things are worth explaining here. The --ipc=host flag and the two --ulimit flags are not optional. Without them, CUDA shared memory operations fail with cryptic errors that are annoying to debug. Always include them.

The model will download on first run (once, thanks to the volume mount), then you'll see vLLM print its startup logs and eventually:

output 
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8000 (Press CTRL+C to quit)

You're live.

Gated Models (Llama, etc.)

If you're pulling a model that requires Hugging Face access approval, pass your token as an environment variable:

bash 
MODEL="meta-llama/Llama-3-8B-Instruct" && docker run -it --gpus all \
  -p 8000:8000 \
  -v ~/.cache/huggingface:/models \
  -e MODEL_ID="$MODEL" \
  -e HF_TOKEN=$HF_TOKEN \
  --name $(basename "$MODEL") \
  --ipc=host --ulimit memlock=-1 --ulimit stack=67108864 \
  vllm-universal

Multi-GPU with Tensor Parallelism

Got a 70B model and four GPUs? Use tensor parallelism to split the model across them:

bash 
MODEL="meta-llama/Llama-3-70B-Instruct" && docker run -it --gpus all \
  -p 8000:8000 \
  -v ~/.cache/huggingface:/models \
  -e MODEL_ID="$MODEL" \
  -e TENSOR_PARALLEL_SIZE=4 \
  --name $(basename "$MODEL") \
  --ipc=host --ulimit memlock=-1 --ulimit stack=67108864 \
  vllm-universal
Heads up: TENSOR_PARALLEL_SIZE must match your actual GPU count. Setting it higher than what's available causes a Ray placement group timeout that isn't immediately obvious. Run nvidia-smi -L first to confirm.

Testing the Endpoint

With the container running, you can talk to it over HTTP. First, confirm which model is loaded:

bash 
curl http://localhost:8000/v1/models

Then send a chat completion request:

bash 
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "Mistral-7B-Instruct-v0.3", "messages": [{"role": "user", "content": "Explain PagedAttention in two sentences."}], "max_tokens": 200, "temperature": 0.7}'

The model name in the request body should match what /v1/models reports — which defaults to the basename of the Hugging Face path. You can override this by setting SERVED_MODEL_NAME.

Streaming Responses

vLLM supports streaming out of the box — no extra configuration required. Add "stream": true to your request and tokens start arriving immediately rather than waiting for the full completion:

bash 
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "Mistral-7B-Instruct-v0.3", "messages": [{"role": "user", "content": "Count to 10."}], "max_tokens": 100, "stream": true}'

The response arrives as a series of data: server-sent events, each containing a partial token delta — the same format as the OpenAI API, so any client library that supports streaming there will work identically here.

Security — Don't Skip This

By default, vLLM starts with no authentication. Anyone who can reach port 8000 can query your model and rack up GPU utilization. This is fine on a laptop behind a firewall, but needs attention the moment you expose the service beyond localhost.

The simplest fix is the built-in --api-key flag. Pass it via EXTRA_ARGS:

bash 
MODEL="mistralai/Mistral-7B-Instruct-v0.3" && docker run -it --gpus all \
  -p 8000:8000 \
  -v ~/.cache/huggingface:/models \
  -e MODEL_ID="$MODEL" \
  -e EXTRA_ARGS="--api-key my-secret-key" \
  --name $(basename "$MODEL") \
  --ipc=host --ulimit memlock=-1 --ulimit stack=67108864 \
  vllm-universal

Callers then pass the key as a Bearer token:

bash 
curl http://localhost:8000/v1/chat/completions \
  -H "Authorization: Bearer my-secret-key" \
  -H "Content-Type: application/json" \
  -d '{"model": "Mistral-7B-Instruct-v0.3", "messages": [{"role": "user", "content": "Hello"}], "max_tokens": 50}'
Never expose port 8000 publicly without authentication. For team deployments, put nginx or Caddy in front of vLLM — terminate TLS there and let the reverse proxy forward requests to the container over localhost. This also lets you assign stable DNS names and manage certs centrally.

Quantization — Cutting VRAM in Half

The QUANTIZATION variable accepts awq, gptq, and squeezellm, but they aren't interchangeable. Here's when to use which:

Format When to use Notes
awq Default choice for most models Best speed/quality tradeoff; wide model support on HuggingFace
gptq When an AWQ version isn't available Slightly slower than AWQ; still widely supported
squeezellm Sparse models only Niche use case; most public models don't use it

AWQ is your default. Search HuggingFace for model-name-AWQ and you'll almost always find a pre-quantized version. A 7B model that normally needs ~16 GB of VRAM drops to around 6–8 GB with AWQ, at the cost of marginal quality degradation that's imperceptible for most tasks.

bash 
MODEL="TheBloke/Mistral-7B-Instruct-v0.2-AWQ" && docker run -it --gpus all \
  -p 8000:8000 \
  -v ~/.cache/huggingface:/models \
  -e MODEL_ID="$MODEL" \
  -e QUANTIZATION=awq \
  --name $(basename "$MODEL") \
  --ipc=host --ulimit memlock=-1 --ulimit stack=67108864 \
  vllm-universal

Debugging Common Errors

Model re-downloads every run

This means the volume isn't mounted correctly. The Dockerfile sets HF_HOME=/models, so the volume must map to exactly /models. If you mount it to /root/.cache/huggingface instead, you'll hit this every time. The correct flag is always -v ~/.cache/huggingface:/models.

CUDA driver error / InductorError

Almost always caused by missing --ipc=host or the ulimit flags. Add them and try again.

docker: invalid reference format

This happens when you paste multi-line commands with backslash continuations directly into your terminal. The shell misparses the image name. The fix is to assign your model to a variable first (MODEL="org/name"), then run the command as a single line.

Out of memory

vLLM pre-allocates KV cache for the entire context length at startup — that's why MAX_MODEL_LEN is the first thing to tune when you hit OOM. A model that fits at 16k context may fail at 32k simply because of this reservation. Try these options in order:

  • Lower MAX_MODEL_LEN — e.g. -e MAX_MODEL_LEN=16384
  • Lower GPU_MEMORY_UTILIZATION — e.g. -e GPU_MEMORY_UTILIZATION=0.80
  • Use a quantized model — -e QUANTIZATION=awq cuts VRAM usage by roughly half
  • Use a smaller model entirely
Tip: Run llmfit to get a quick estimate of which models fit your current GPU configuration without needing to actually attempt loading them.

Performance tuning

Once you have a model running, two flags are worth knowing about for squeezing more throughput. --enable-chunked-prefill splits long prefills across multiple forward passes, which reduces latency spikes on long prompts. --max-num-batched-tokens controls how many tokens are processed in a single batch — raising it helps throughput under high concurrency. Pass both via EXTRA_ARGS:

bash 
-e EXTRA_ARGS="--enable-chunked-prefill --max-num-batched-tokens 8192"

Docker Compose Setup

If you're already running a stack with Docker Compose, dropping vLLM in as a service is straightforward. Create a docker-compose.yml:

docker-compose.yml 
services:
  vllm:
    image: vllm-universal
    ports:
      - "8000:8000"
    volumes:
      - ~/.cache/huggingface:/models
    environment:
      - MODEL_ID=mistralai/Mistral-7B-Instruct-v0.3
      - MAX_MODEL_LEN=16384
      - GPU_MEMORY_UTILIZATION=0.85
      - EXTRA_ARGS=--api-key ${VLLM_API_KEY}
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
    ipc: host
    ulimits:
      memlock: -1
      stack: 67108864
    restart: unless-stopped

Then bring it up with:

bash 
VLLM_API_KEY=my-secret-key docker compose up -d
Tip: Store VLLM_API_KEY in a .env file alongside your compose file. Docker Compose picks it up automatically and keeps secrets out of your shell history.

Useful Commands to Keep Handy

bash 
# Check available GPUs
nvidia-smi -L

# Monitor GPU utilization while model loads
nvtop

# Check available disk space (models are large)
df -h

llmfit

docker ps

# Stream container logs
docker container logs -f <container_id>

# Stop and remove a container
docker stop <name> && docker rm <name>

Wrapping Up

That's really the whole setup. One Dockerfile, one build, and from that point on hosting a new model is a single command. The volume mount means model weights survive container restarts, and the OpenAI-compatible API means you don't have to rewrite anything to switch from a cloud provider to your local GPU.

Where you go from here depends on your use case. For team environments, lock the endpoint down with --api-key and put a reverse proxy in front. For inference at scale, continuous batching and streaming are on by default — no extra configuration needed. For constrained hardware, reach for an AWQ-quantized model first before trying anything else. In the next post, we'll take this local vLLM endpoint a step further and wire it directly into OpenClaw — so your AI agent has a fully private, self-hosted brain to reason with.

For now though — if curl localhost:8000/v1/models returns your model name, you're in business.
