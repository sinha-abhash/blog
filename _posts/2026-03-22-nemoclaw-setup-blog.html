Running OpenClaw Inside NemoClaw

March 2026 · 25 min read · Requires: vLLM running on :8000

Running OpenClaw bare on the host gets you up and running fast — but the agent has broad access to your filesystem and network by default. NemoClaw changes that. It wraps OpenClaw in a Landlock + seccomp sandbox with explicit network policies, while keeping your local vLLM inference intact. This is the field guide for getting there — including everything that went wrong along the way.

What NemoClaw Actually Does

NemoClaw is NVIDIA's open-source security wrapper around OpenClaw. It runs your agent inside an OpenShell sandbox enforced by Landlock, seccomp, and network namespaces — meaning the agent can only touch approved filesystem paths and approved network endpoints. Everything else is denied at the kernel level.

Your existing OpenClaw installation outside the sandbox is completely untouched. Both run side by side — bare OpenClaw for quick tasks, sandboxed OpenClaw when the boundary matters.

Early Preview: NemoClaw is not production-ready as of March 2026. Interfaces and APIs may change without notice. Treat this as a field report, not stable documentation.

Before You Start

You'll need the following in place before anything else:

  • Ubuntu 22.04+ — required for Landlock kernel support
  • Node.js v20+ — v22 recommended
  • Docker running — NemoClaw runs k3s inside Docker
  • OpenShell — install via GitHub releases
  • 16 GB RAM recommended — sandbox image is ~2.4 GB compressed
  • 40 GB free disk recommended

Install Node.js v22 and OpenShell:

bash 
# Install Node.js v22
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs

# Install OpenShell
ARCH=$(uname -m) && gh release download   --repo NVIDIA/OpenShell   --pattern "openshell-${ARCH}-unknown-linux-musl.tar.gz" && tar xzf openshell-${ARCH}-unknown-linux-musl.tar.gz && sudo install -m 755 openshell /usr/local/bin/openshell

Install NemoClaw and Onboard

A single installer script handles everything — the sandbox image, the gateway, and the guided onboard wizard:

bash 
curl -fsSL https://nvidia.com/nemoclaw.sh | bash

The wizard runs through 7 steps. At step 4 it asks for an NVIDIA API key to configure the NIM inference provider. If you're planning to use local vLLM (which we are), you don't need a real key — but the wizard validates the format and rejects anything that doesn't start with nvapi-.

Key format gotcha: Entering a plain dummy value like dummykey will be rejected with "Invalid key. Must start with nvapi-". Use nvapi-dummykey to pass validation. You'll switch to vllm-local immediately after onboarding anyway.

Step 4/7: API key prompt accepts the placeholder, then lists available cloud models — select GPT-OSS 120B (option 6) or any model, since we'll switch providers right after.

At step 7, the wizard shows available network policy presets. Use Space to toggle and Enter to confirm. Select at minimum npm and pypi — add huggingface if you work with models directly.

Step 7/7: Policy presets. The wizard suggests pypi and npm — accept both. You can add more later with nemoclaw dtx-claw policy-add.

When complete, you'll see a summary like:

output 
Sandbox      dtx-claw (Landlock + seccomp + netns)
Model        openai/gpt-oss-120b (NVIDIA Cloud API)
──────────────────────────────────────────────────
Run:         nemoclaw dtx-claw connect
Status:      nemoclaw dtx-claw status
Remember your sandbox name. Whatever you chose during onboarding — dtx-claw in our case — must be used in every NemoClaw command. The docs show my-assistant as an example, but that's not your sandbox name unless you actually typed it.

Point NemoClaw at Your Local vLLM

By default NemoClaw uses the NVIDIA cloud API which incurs per-token costs. To run entirely on your own GPU, you need to create a local provider and switch to it. All five steps run on the host, not inside the sandbox.

Step 1 — Find your exact model name

bash 
curl http://localhost:8000/v1/models

Note the id field exactly. vLLM may report gpt-oss-120b even when started with openai/gpt-oss-120b. Always use what the API reports, not what you passed at startup.

Step 2 — Create the vllm-local provider

Known bug: vllm-local is not registered during onboarding. It must be created manually — and recreated any time you destroy and re-onboard the sandbox.
bash 
openshell provider create   --name vllm-local   --type openai   --credential OPENAI_API_KEY=dummy   --config OPENAI_BASE_URL=http://host.openshell.internal:8000/v1

Use host.openshell.internal — never localhost. Inside the sandbox's network namespace, localhost resolves to the sandbox itself, not your host machine.

Step 3 — Open the firewall

bash 
sudo ufw allow proto tcp from 172.18.0.0/16 to any port 8000 comment 'OpenShell cluster to vLLM'
sudo ufw allow proto tcp from 172.17.0.0/16 to any port 8000 comment 'Docker bridge to vLLM'

Step 4 — Switch the active provider

bash 
openshell inference set --provider vllm-local --model <id-from-step-1>

Step 5 — Verify

bash 
openshell inference get
# Expected: Provider: vllm-local, Model: gpt-oss-120b
Best confirmation: Watch your vLLM server logs while chatting in OpenClaw. Each response should produce a POST /v1/chat/completions HTTP/1.1" 200 OK line. If you don't see traffic, the provider switch didn't take — recheck with openshell inference get.

vLLM server logs confirming each chat message routes through local inference. This is the most reliable way to verify the provider switch actually took.

Accessing the Web UI

NemoClaw's gateway serves a full browser-based interface for OpenClaw. Connect to the sandbox first, then generate the authenticated URL:

bash 
# Connect to sandbox
nemoclaw dtx-claw connect

# Get dashboard URL with embedded auth token
openclaw dashboard --no-open

Copy the printed URL and open it directly in your browser — it includes the auth token as a query parameter so no login is required.

Never navigate directly to the port. The Web UI auto-retries without a valid token and hits a rate limit fast. Once locked out, you have to wait for the cooldown. Always use openclaw dashboard --no-open to generate a pre-authenticated URL.

OpenClaw's Web UI running inside the NemoClaw sandbox — inference routed through local vLLM, no cloud API involved.

Editing Config Inside the Sandbox

OpenClaw's config inside the sandbox lives at /sandbox/.openclaw/openclaw.json — but this path is Landlock read-only. You cannot write there directly. There are two ways around it.

Option A — Use the CLI

bash 
# Read a value
openclaw config get <dot.path>

# Set a value
openclaw config set <dot.path> <value>

# Validate
openclaw config validate

Option B — Copy to a writable location

bash 
# Copy config to writable path
cp /sandbox/.openclaw/openclaw.json /sandbox/.openclaw-data/openclaw.json

# Point OpenClaw at the writable copy
echo 'export OPENCLAW_CONFIG_PATH=/sandbox/.openclaw-data/openclaw.json' >> ~/.bashrc
source ~/.bashrc
Note: The Web UI config editor uses JSON5 syntax — unquoted keys and trailing commas are valid. This is intentional, not a bug.

Things Nobody Warns You About

vllm-local not auto-registered

The docs say to run openshell inference set --provider vllm-local but the provider doesn't exist until you create it manually. Always run openshell provider create first — and again after every destroy.

Use host.openshell.internal, not localhost

Inside the sandbox's network namespace, localhost resolves to the sandbox itself. Any provider URL pointing at localhost will silently fail to reach your host's vLLM server.

destroy wipes everything — and there's no stop

nemoclaw dtx-claw destroy deletes the sandbox and all data inside it — including your custom provider config. There is no stop or start. If you just want to detach from the sandbox temporarily, close the terminal or press Ctrl+D — the sandbox keeps running in the background. Valid actions are connect, status, logs, policy-add, policy-list, and destroy.

The gateway runs on the host, not in the sandbox

Running openclaw gateway restart inside the sandbox has no effect on the actual running gateway. To reload it, reconnect from the host with nemoclaw dtx-claw connect.

File extraction via base64

kubectl cp and openshell cp may fail with TLS errors. The reliable method: base64 /path/to/file inside the sandbox, then decode on the host with echo "..." | base64 -d > filename.

npm policy ≠ npm works

Selecting the npm network policy during onboarding doesn't automatically unblock all npm traffic. The binary path in the policy must exactly match the binary making the outbound call. Check nemoclaw dtx-claw logs for CONNECT action=deny entries if npm installs are silently failing.

Automating Restarts with start-ai.sh

After a machine reboot, you need to bring up vLLM, and then reconnect NemoClaw — and recreate the vllm-local provider every time since it gets wiped on each onboard. Doing this manually gets tedious fast.

The start-ai.sh script handles the NemoClaw side in one command. For NemoClaw specifically, it:

  • Frees port 18789 if anything is squatting on it
  • Kills any stale openshell gateway on port 8080
  • Runs nemoclaw onboard and auto-answers the "Recreate?" prompt
  • Recreates the vllm-local provider automatically
  • Switches inference to vllm-local / gpt-oss-120b
  • Connects to the sandbox and attempts to print the dashboard URL
bash 
./start-ai.sh <vllm-container-id>
Usage: Pass your vLLM Docker container ID as the only argument. The script confirms the container exists, waits for vLLM to be ready, then walks through each phase interactively — asking whether to start host OpenClaw and NemoClaw separately.

Post-Install: File Explorer Crash

After completing the NemoClaw installation, you may find that Nautilus silently refuses to open. Running it in a terminal reveals a segfault:

output 
(org.gnome.Nautilus:142157): Adwaita-WARNING **: No known Yaru accent 'nvidia'
Segmentation fault (core dumped)

The NemoClaw installer sets the GTK theme to Yaru-nvidia-dark and configures NVIDIA GPU acceleration hooks. GTK4 tries to use those hooks, fails, and crashes. Confirm it's the cause:

bash 
gsettings get org.gnome.desktop.interface gtk-theme
# Returns: 'Yaru-nvidia-dark'  ← the culprit

Fix it by forcing GTK4 to use software rendering permanently:

bash 
# Create the environment config
mkdir -p ~/.config/environment.d
echo 'GSK_RENDERER=cairo' > ~/.config/environment.d/gsk-renderer.conf

# Apply to the current live session without logout
dbus-update-activation-environment --systemd GSK_RENDERER=cairo
systemctl --user set-environment GSK_RENDERER=cairo

# Relaunch Nautilus
pkill -9 nautilus && sleep 1 && nautilus &
Confirmed working: After a full reboot, Nautilus opens normally from the dock. The ~/.config/environment.d/ file ensures the variable is set on every login. This fix is entirely independent of your OpenClaw and NemoClaw installation — the sandbox doesn't use GTK.

If your browser went dark during troubleshooting after switching the GTK theme, restore it with:

bash 
gsettings set org.gnome.desktop.interface color-scheme 'default'

Daily Driver Commands

bash 
# Connect to sandbox
nemoclaw dtx-claw connect

# Get Web UI URL with embedded auth token
openclaw dashboard --no-open

# Check sandbox status and gateway port
nemoclaw dtx-claw status

# Stream sandbox logs
nemoclaw dtx-claw logs --follow

# Check active inference provider
openshell inference get

# Switch to local vLLM
openshell inference set --provider vllm-local --model <model-id>

# Add a network policy
nemoclaw dtx-claw policy-add

# Extract a file from the sandbox
base64 /path/to/file          # inside sandbox
echo "..." | base64 -d > file # on host

Wrapping Up

At this point OpenClaw is running inside a proper sandbox — Landlock filesystem restrictions, seccomp syscall filtering, and network access locked to an explicit allowlist. Your vLLM server connects in via host.openshell.internal, keeping all inference local while the agent operates under meaningful constraints.

The parts that bite most people: vllm-local not existing until you create it, localhost silently resolving to the sandbox, and the Web UI lockout from navigating to the bare port. All documented above.

If you haven't set up bare OpenClaw with a local vLLM model first, the previous post covers that foundation — this guide builds directly on top of it.
