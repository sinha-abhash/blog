Wire OpenClaw to Your Local vLLM Model

March 2026 · 12 min read · Requires: vLLM running on :8000

You've got a vLLM server humming along on port 8000 — a capable model, low latency, no API bills. Now you want OpenClaw, the open-source coding agent, to use that model as its brain instead of phoning home to a cloud provider. This guide walks through exactly that: pointing OpenClaw at your local inference endpoint, whether you're setting it up for the first time or reconfiguring an existing install.

How OpenClaw Connects to a Model

OpenClaw uses a provider abstraction — a named configuration entry that maps an identifier to a base URL and credentials. When you ask it to reason or generate code, it looks up the active provider, constructs an OpenAI-compatible API call, and fires it at whatever URL that provider points to.

Since vLLM speaks the OpenAI API natively, the wiring is conceptually simple: tell OpenClaw to use vLLM as its provider, give it your server's base URL, and specify which model to request. OpenClaw has a guided configure wizard that handles all of this interactively — no manual config editing required.

Assumption: Your vLLM server is already up and responding at http://localhost:8000. Run curl http://localhost:8000/v1/models before continuing — if that returns a model list, you're ready.

Find Your Exact Model Name First

Before touching the OpenClaw config, confirm the exact model name that vLLM is serving. This sounds trivial but it's the source of a frustrating 404 error later if you skip it.

If you started vLLM with a path like openai/gpt-oss-120b, it may report the model as gpt-oss-120b — the basename without the org prefix. Always check:

bash 
curl http://localhost:8000/v1/models

Look at the id field in the response. That exact string is what you'll enter when the wizard asks for the model name. Write it down.

Heads up: The id from /v1/models is the ground truth — not what you passed to vllm serve. Use what the API actually reports.

Path A — Configure During Initial Setup

When you run OpenClaw for the first time, the setup wizard walks you through provider selection as part of the onboarding flow. At the Model/auth provider step, scroll down and select vLLM — it's listed near the bottom as Local/self-hosted OpenAI-compatible.

After selecting vLLM, the wizard immediately prompts for three things in sequence: the base URL of your vLLM server, the API key (enter empty if your server has no auth), and the model name — use the id you found in step 2.

Once you confirm, the wizard shows Model configured and sets the default model to vllm/gpt-oss-120b. That's it for the initial path — skip ahead to section 5 to verify the connection.

Base URL format: Enter the full path including /v1 — for example http://127.0.0.1:8000/v1. OpenClaw appends endpoint paths like /chat/completions on top of this.

Path B — Reconfigure an Existing Install

Already have OpenClaw installed and want to switch it to your local vLLM server? The same guided wizard is available at any time via:

bash 
openclaw configure

The wizard opens and shows your existing config at the top — workspace path, current model, gateway settings. It then asks where the gateway will run: choose Local (this machine).

Next it presents a list of sections to configure. Select Model (Pick provider + credentials).

From here the flow is identical to Path A — select vLLM as the provider, enter the base URL, API key, and model name. The wizard confirms with Model configured when done.

Don't select Continue without saving. Make sure you complete the Model section and see the Model configured confirmation before selecting Continue, otherwise your changes won't be written.

Verify the Connection Is Actually Working

Config looking correct is not the same as requests actually reaching your vLLM server. The most reliable way to verify is to watch the vLLM logs while sending a request from OpenClaw.

Open a second terminal and tail the vLLM output. If you started it with Docker:

bash 
docker container logs -f <container_name>

Now trigger a request from OpenClaw — even a simple prompt works. In the vLLM log window you should see:

output 
INFO:     127.0.0.1 - "POST /v1/chat/completions HTTP/1.1" 200 OK

Every OpenClaw response should produce one of these lines. No traffic at all means OpenClaw isn't reaching your server — recheck the base URL. A 404 means the model name is wrong — go back to step 2.

Best confirmation: Watching the vLLM server log light up on each OpenClaw response is the definitive proof. Don't trust the config output alone — trust the traffic.

Accessing the Web UI

OpenClaw ships with a browser-based interface. Getting to it requires one non-obvious step — and skipping it will get you locked out.

Don't navigate directly to the port in your browser. The Web UI expects an auth token embedded in the URL. Navigating without it triggers repeated failed authentication attempts, and after too many failures OpenClaw imposes a cooldown lockout.

Instead, always generate the URL this way:

bash 
openclaw dashboard --no-open

This prints a URL with the auth token already embedded as a query parameter. Copy it and paste it directly into your browser:

output 
http://127.0.0.1:18789/?token=eyJhbGciOiJIUzI1NiIsIn...
Never open the bare port URL directly. Bookmarking localhost:18789 will cause lockouts. Always generate a fresh URL with openclaw dashboard --no-open.

Debugging Common Errors

HTTP 404 — model does not exist

The model name entered during setup doesn't match what vLLM is actually serving. Run curl http://localhost:8000/v1/models, copy the id field exactly, and re-run openclaw configure to correct it. Case-sensitive — character-for-character match required.

Requests aren't reaching vLLM at all

Re-run openclaw configure and verify the base URL is exactly http://127.0.0.1:8000/v1 (including /v1). If the URL looks right but traffic still isn't showing in vLLM logs, check whether a firewall like UFW is blocking port 8000 — some configurations block loopback traffic from certain processes even on localhost.

Unauthorized: gateway token mismatch

You navigated to the Web UI port directly, without the embedded auth token. Run openclaw dashboard --no-open and use the printed URL instead.

Unauthorized: too many failed authentication attempts

The Web UI auto-retried without a valid token and hit the rate limit. Wait a few minutes for the lockout to clear, then use openclaw dashboard --no-open. Don't try to force-navigate while locked out — it resets the cooldown timer.

Useful Commands to Keep Handy

bash 
# Confirm the exact model name vLLM is serving
curl http://localhost:8000/v1/models

# Run the interactive setup wizard (first time or reconfigure)
openclaw configure

# Get Web UI URL with embedded auth token
openclaw dashboard --no-open

# Watch vLLM traffic to confirm requests are arriving
docker container logs -f <container_name>

Wrapping Up

That's the full setup. Whether you wired in vLLM during the initial wizard or switched to it later via openclaw configure, the end result is the same: OpenClaw running entirely on your hardware, no cloud calls, no token costs, no data leaving the machine.

The parts that trip people up most are almost always the same two: using the wrong model name because vLLM's served name differs from what you passed at startup, and navigating to the Web UI port directly and triggering a lockout. Both are easy fixes once you know they're coming.
